Le processus de développement

La construction de logiciel en entreprise peut s'apparenter à une chaîne de montage.

C'est une succession d'activités qui mènent aux serveurs de production.

Généralement, on peut découper cette chaîne en 3 parties.

En pratique, cela peut prendre la forme d'un V, d'une cascade, d'une spirale ou bien d'itérations courtes selon la méthodologie suivie.

Nous nous intéresserons principalement à l'étape développement.

Les outils du développement

Voici une vision globale des outils pouvant participant à l'activité de développement.

L'écriture du code source se fait à l'aide d'un IDE. Nous utiliserons eclipse. Netbeans et intellij sont d'autres IDE très complets.

Pour le versionning, subversion est l'outils le plus utilisé. Cependant, git (notamment via github) gagne rapidement en popularité.

Un outil de build a pour but la création du produit final à partir du code source. L'outil de build classique pour java est maven. Des alternatives sont ant, ivy ou gradle.

Le repository (dépot en français) permet de stoquer les builds. Nexus et artifactory sont des solutions qui s'intègrent parfaitement avec maven.

Nous nous intéresserons à l'écriture du code et au build.

L'intégration continue

L'intégration continue est un pratique qui vise à tester le bon déroulement des builds aussi souvent que possible : potentiellement après chaque commit.

Cela permet :

  • D'être informé d'un problème le plus tôt possible.
  • D'éviter les effets big bang de fin de projet.

En plus du build, ce processus automatisé est aussi l'occassion de générer des rapports sur le code afin d'en améliorer la qualité.

L'outils open source de référence est jenkins

Des plugins ou outils complémentaires comment pmd ou sonar permettent d'obtenir des métriques sur le code.

Le déploiement continu

Le déploiement continu est une extension de l'intégration continue : en plus d'intégrer régulièrement, on déploit en production régulièrement.

Voici la chaîne en place chez RTL net :

  • Les demandes sont faites sous jira
  • Les développeurs répondent aux demandes en développant
  • Jenkins s'occupe de l'intégration continu
  • Rundeck déploit en production quand une version est prête
As A Service

Avec la standardisation des outils et l'apparition des offres de type cloud, il existe des solutions SAAS pour le développement.

Pour le versionning, il est existe des solutions comme assembla ou github.

Pour le build, cloudbees par exemple propose une solution d'intégration continue avec jenkins.

Coté hébergement (PAAS), il existe aussi de nombreuses solutions :

  • Google App Engine
  • Cloud Foundry
  • Cloudbees
