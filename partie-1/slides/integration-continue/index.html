Intégration continue

L'intégration

Réunir toutes les pièces afin de construire le logiciel à installer s'appelle l'intégration.

Une intégration peut varier de quelques secondes à plusieurs jours.

En cas d'erreur lors de l'intégration, le coût peut donc être lourd.

En cas de bug en production, une intégration lente empêche une correction rapide.

On recommande donc une intégration courte et automatisée.


Sur un projet maven isolé, cela se résume à une commande :

mvn package

Sur des projets complexes, le nombre d'étapes peut être élevé.

Une pratique agile

L'intégration continue est une pratique mise en avant par les méthodologies agiles.

Le principe est de réaliser des intégrations aussi fréquemment que possible.

If it hurts, do it more often

Kent Beck

La fréquence optimale est de faire après chaque commit.

Le but est de détecter l'apparition d'anomalies le plus tôt possible.

Quand le build passe en échec, c'est que l'erreur vient d'être introduite dans le code.

Bénéfices

Les bénéfices de l'intégration continue sont multiples :

  • Responsabilisation des commiters
  • Détection rapide d'une modification qui casse le build
  • Moins de surprises en fin de projets (effet big bang)
  • Permet de déployer en production plus souvent
  • Permet de surveiller l'état d'avancement d'un projet (effet tunnel)

Un indicateur de santé de l'intégration continue est souvent installé au sein des équipes.

Pratiques clés

Voici quelques pratiques associées à l'intégration continue:

  • Un seul VCS pour le code source
  • Le build est automatisé (rien de manuel)
  • Le build exécute les tests unitaires
  • Tout le monde commit quotidiennement dans la branche maitre
  • Tout le monde peut voir le résultat des builds
  • Chaque commit dans la branche maitre déclenche un build
  • Le build doit être rapide
  • Les livrables sont disponibles facilement

http://martinfowler.com/articles/continuousIntegration.html

Cette liste n'est pas exhaustive mais définit un état d'esprit lié au build.

Dans la continuité

Si on intègre en continu, pourquoi ne pas mettre en prod dans la foulée ?

Le déploiement continue vise à mettre en prod le plus souvent possible.

Voici la chaîne en place chez RTL net :

  • Les demandes sont faites sous jira
  • Les développeurs répondent aux demandes en développant
  • Jenkins s'occupe de l'intégration continue
  • On déploit via Rundeck quand une version est prête
